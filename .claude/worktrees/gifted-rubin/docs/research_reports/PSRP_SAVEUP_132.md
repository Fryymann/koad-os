## PSRP Saveup — Issue #132 — 2026-03-10

### 1. Fact (What happened?)
I implemented the Sentinel v4 Hydration Protocol to support token-less context loading. I updated the `HotContextChunk` schema across `koad-proto` and `koad-core` to include a `file_path` field and refactored the `KoadHydrationManager` in `koad-spine` to handle reference-based hydration. When a file path is provided via the CLI (`koad system context hydrate --path`), the Spine now independently assesses the file: small files are ingested fully, while large files generate a "Virtual Chunk" (metadata/summary). I verified this by hydrating `Cargo.lock`, which resulted in a 106-byte summary instead of a 100KB+ token dump.

### 2. Learn (Why did it happen / What is the underlying truth?)
I learned that the "Cognitive Operating System" must act as a gatekeeper for the model's limited context window. By moving the "Read" operation from the Client (the model's terminal) to the Spine (the OS background), we decouple "Discovery" from "Reasoning." The model no longer needs to see the *entire* file to know the file *exists* and what it *contains*. This fundamental shift allows KoadOS to scale to repositories that would otherwise exceed LLM token limits or budgets. I also learned that schema updates in a gRPC-heavy system require surgical precision to avoid breaking initializers in disconnected modules like `import.rs` or `test_client.rs`.

### 3. Ponder (How does this shape future action?)
This implementation is the bedrock for the "Local Micro-Agent Tier." In the future, when a Virtual Chunk is created for a large file, we could trigger a local Ollama model to generate a high-signal architectural summary of that file, which then hydrates the primary agent's context. We have moved from "Raw Data Injection" to "Intelligent Signal Propagation." Moving forward, we should extend this to entire directories, allowing an agent to "hydrate a project" by simply passing a root path, with the Spine generating a comprehensive "Context Map" automatically.
