
# ASM & Spine Reliability Review — KoadConfig Integration

**Date:** 2026-03-11

**Reviewer:** Tyr

**Authority:** Dood Override

## Executive Summary

The KoadOS Agent Session Manager (ASM) and Spine architecture have evolved into a distributed system using Redis as the authoritative hot-path state. While functional, the system exhibits **non-atomic race conditions** during the boot sequence and relies heavily on **hardcoded string matching** for role-based security and session policy.

This review identified that while the infrastructure for a robust, config-driven OS is present via `KoadConfig`, the core logic of session integrity (Heartbeats, Pruning, and Sandboxing) has not yet been fully decoupled from the original human-centric design. The primary risks include cognitive leakage during concurrent boots and session "zombification" where stale environment variables or orphaned heartbeat daemons prevent clean system re-entry.

**Top Recommendations:**
1. **Atomic Lease Scripting:** Implement a Redis Lua script for `acquire_lease` to prevent concurrent boot collisions.
2. **Config-Driven Sandboxing:** Refactor the `Sandbox` to use `Rank` and `Permissions` from `IdentityConfig` instead of hardcoded strings like "admin" or "captain".
3. **Tiered Session Policies:** Allow per-agent overrides for `SessionsConfig` (e.g., higher-tier agents like Tyr require longer purge timeouts for deep-grid operations).

## 1. Architecture Audit

### 1.1 Boot Sequence (`koad boot`)

- **Resolution:** Agent name is parsed from CLI args. Identity configuration is loaded from `identities/<agent>.toml` or falls back to legacy `config/` lookups.
- **SID/BodyID Generation:** `session_id` is generated as a UUID by the Spine. `body_id` is generated by the CLI or reused from the environment.
- **Race Conditions:** `handle_boot_command` (CLI) and `acquire_lease` (Spine) both perform an `HGET` then `HSET` sequence in Redis. This is **UNVERIFIED ATOMICITY**. Concurrent boots can theoretically result in two active sessions for the same sovereign agent if they clear the `HGET` check simultaneously. (Ref: `crates/koad-spine/src/engine/identity.rs:52`)
- **Initialization Order:** 1. CLI Pre-flight -> 2. gRPC `InitializeSession` -> 3. Spine Lease Acquisition -> 4. Redis `koad:state` update -> 5. CLI Heartbeat/Watchdog spawn.

### 1.2 Body/Ghost Tethering

- **Mechanism:** Environment variables `KOAD_SESSION_ID` and `KOAD_BODY_ID` bind the Body (terminal shell) to the Ghost (Identity + Hot Context).
- **Enforcement:** Spine `prune_body_ghosts` pre-emptively marks old sessions on the same body/driver as `dark` during initialization. (Ref: `crates/koad-spine/src/engine/asm.rs:216`)
- **Sovereign Locking:** Sovereign agents (Tyr, Koad, Dood) are locked to a single session via the `koad:kai:<Name>:lease` Redis key.
- **Failure Modes:** If a shell crashes without `koad logout`, the heartbeat daemon may become orphaned, maintaining the session as `active` in Redis. Re-entry requires the `--force` flag to prune the stale lease.

### 1.3 Agent Session Manager (ASM)

- **Tracking:** Authoritative state lives in Redis `koad:state`. Both the ASM daemon (`koad-asm`) and the Spine's internal ASM module sync their local `HashMap` caches from this hash.
- **Reaping Logic:** The `koad-asm` daemon is the primary reaper. It marks sessions `dark` after `dark_timeout_secs` (default 60s) and purges after `purge_timeout_secs` (default 300s). (Ref: `crates/koad-asm/src/main.rs:114`)
- **Deadman Switch:** Tier 1 agents (Captain+) trigger an emergency `drain_all` to SQLite if they flatline for more than 45s (`deadman_timeout_secs`).

### 1.4 Spine Connectivity

- **Coordination:** Coordination between the ASM daemon and Spine is handled via the `koad:sessions` Redis PubSub channel.
- **Sentinel (Diagnostics):** The `ShipDiagnostics` loop (running every 5s) performs "Autonomic Recovery" for crashed services like `koad-asm` and `kgateway`. (Ref: `crates/koad-spine/src/engine/diagnostics.rs:347`)
- **State Consistency:** If the Spine restarts, it re-hydrates its local session cache from Redis. If Redis crashes, state is restored from SQLite `state_ledger` by the Sentinel's `check_registry_integrity` logic.

## 2. Failure Analysis

### 2.1 Cognitive Mismatch Scenarios

- **Concurrent Boot:** Two sessions for the same agent can exist if the `HGET/HSET` race window is hit. This leads to interleaved context in Redis keys like `koad:context:session:<SID>`.
- **Inherited SID:** If a new shell inherits a `KOAD_SESSION_ID` from a parent but the agent has changed (e.g., `koad boot --agent Pippin` while `Tyr` was active), the CLI correctly warns of a collision, but if bypassed, it could lead to identity drift.

### 2.2 Session Integrity Failures

- **Orphaned Heartbeats:** Heartbeat daemons spawned with `spawn()` (Ref: `crates/koad-cli/src/handlers/boot.rs:188`) may persist after the shell is closed, keeping a "Zombie Ghost" alive in the Spine.
- **Redis Overwrite:** If two different bodies boot the same agent (Sovereign) with `--force`, the first body loses its lease but its `KOAD_SESSION_ID` environment variable remains, potentially allowing it to continue sending commands to a "Dark" session.

### 2.3 Recovery Gaps

- **Hot Context Loss:** While session metadata is drained to SQLite every 30s, the "Hot Context" (conversation history) is only quicksaved every 5 minutes. A Redis crash results in a loss of all context injected since the last quicksave. (Ref: `crates/koad-spine/src/engine/diagnostics.rs:649`)
- **Lease Persistence:** If the Spine crashes, leases are preserved in Redis. However, if the `koad-asm` reaper also fails, there is no mechanism to clear these leases until a manual `--force` boot.

## 3. KoadConfig Integration Recommendations

### 3.1 Session Policy (TOML Overrides)

- **Recommendation:** Migrate `SessionsConfig` (timeouts/intervals) to be overridable per-agent in `identities/*.toml`.
- **Benefit:** High-availability agents (Tyr) can have longer timeouts to prevent premature purging during complex computations, while transient agents (Crew) have aggressive purging.

### 3.2 Identity Binding (Canonical Source)

- **Recommendation:** Replace legacy `config/` lookups in `koad-cli/main.rs` with `KoadConfig` as the primary source of truth for agent bios and roles.
- **Benefit:** Eliminates "Identity Drift" where the CLI and Spine disagree on an agent's rank or bio.

### 3.3 Cognitive Isolation (Namespacing)

- **Recommendation:** Derive Redis key namespacing (e.g., `koad:context:<agent_name>:<SID>`) from configuration.
- **Benefit:** Reduces the impact of SID collisions and makes context hydration more verifiable against the agent's declared profile.

### 3.4 Spine Resilience (Config-Driven Recovery)

- **Recommendation:** Move `drain_interval` and `reaper` thresholds from hardcoded constants to `kernel.toml`.
- **Benefit:** Allows fine-tuning of system performance and durability without requiring a re-compile of the Spine.

### 3.5 Migration Priority

1. **High:** `Sandbox` roles (from hardcoded strings to `IdentityConfig.rank` mappings).
2. **Medium:** `SessionsConfig` overrides per-agent.
3. **Low:** `NetworkConfig` (ports/sockets) unification across all crates.
