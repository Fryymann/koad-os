## Summary
KoadOS uses SQLite as its primary "cold path" storage for long-term, persistent agent memory. The main database, `cass.db`, is managed by the `koad-cass` service. It is designed to be a durable, queryable archive of all high-signal knowledge and session summaries generated by the agent swarm, forming the foundation of the system's compounding memory.

## How It Works
The entire SQLite database interaction is abstracted behind a `Storage` trait defined in `crates/koad-cass/src/storage/mod.rs`. This is a key architectural decision that allows for easy testing and potential future database backends.

The default implementation is the `CassStorage` struct, which uses the `rusqlite` crate to interact with the `cass.db` file.

The database schema is simple and focused on two main tables:
1.  **`facts`**:
    - Stores `FactCard` data.
    - Columns include `id`, `source_agent`, `session_id`, `domain`, `content`, `confidence` (the AI-generated significance score), `tags`, and `created_at`.
    - This table is the "long-term memory" of the system.

2.  **`episodes`**:
    - Stores `EpisodicMemory` data, which are the `EndOfWatch` summaries.
    - Columns include `session_id`, `project_path`, `summary` (the AI-generated text), `turn_count`, and `timestamp`.
    - This table provides a historical record of what happened during each session.

When a CASS service (like `MemoryService`) needs to interact with the database, it doesn't call `rusqlite` functions directly. Instead, it calls methods on the `storage` object (which is an `Arc<dyn Storage>`). For example, `storage.commit_fact(fact)` or `storage.record_episode(episode)`.

This design has several advantages:
- **Decoupling:** The gRPC services don't need to know about SQL. They just interact with a simple Rust trait.
- **Testability:** In unit tests, the real `CassStorage` can be replaced with a `MockStorage` that uses an in-memory `Vec` instead of a database file, making tests fast and self-contained.
- **Maintainability:** All SQL queries are centralized in one file (`storage/mod.rs`), making the schema easy to manage and migrate.

## Raw Technical Notes
- The choice of SQLite aligns with the "local-first" principle of KoadOS. It's file-based, requires no separate server process, and is extremely portable.
- While Redis is used for "hot" data (like active sessions), SQLite is used for "cold," archival data that needs to persist forever and be queryable.
- Future phases may introduce other storage backends (like `Qdrant` for vector search) which would also be implemented behind the `Storage` trait to maintain architectural consistency.
