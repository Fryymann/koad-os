# The Body/Ghost Model

> The conceptual model that separates an agent's ephemeral shell session (the Body) from its persistent identity and memory (the Ghost), enabling long-lived agents that survive reboots.

**Complexity:** basic
**Related Articles:** [The Tri-Tier Model](./tri-tier-model.md), [Agent Session Lifecycle](../core-systems/agent-session-lifecycle.md), [koad-agent boot](../core-systems/koad-agent-boot.md)

---

## Overview

In KoadOS, every agent has two aspects: a **Ghost** and a **Body**. The Ghost is who the agent *is* — its identity, rank, accumulated memory, and learned knowledge. The Body is where the agent *lives right now* — the active shell session it's currently operating in. The Body is temporary; the Ghost is permanent.

This distinction solves a practical problem: AI agent sessions die. Terminals crash, network connections drop, machines reboot. Without a model that separates persistent identity from ephemeral execution, every session termination would mean losing everything the agent had learned. The Body/Ghost model gives agents continuity across sessions.

The metaphor is intentional. The Ghost is the soul — it persists between sessions, carries memory, and defines capability. The Body is the physical vessel the Ghost inhabits during a working session. When the Body is destroyed (session ends), the Ghost remains intact, ready to inhabit a new Body on the next boot.

This model also enables the **"One Body, One Ghost"** protocol: each named agent (Tyr, Scribe, Sky, etc.) may only have one active Body at any time. The Citadel enforces this at the `CreateLease` level, preventing the same agent identity from being booted in two shells simultaneously — which would create conflicting state and race conditions on shared resources.

## How It Works

### The Ghost (Identity & Memory)

A Ghost consists of two parts: a static identity definition and a dynamic, growing memory store.

**Identity** is defined in `config/identities/<agent_name>.toml`. This TOML file specifies the agent's name, rank (`Admiral`, `Captain`, `Officer`, `Crew`), permissions, and capability tier. The `Identity` struct in `koad-core` represents this at runtime:

```rust
pub struct Identity {
    pub name: String,      // "Tyr"
    pub rank: Rank,        // Rank::Captain
    pub permissions: Vec<String>,
    pub access_keys: Vec<String>,
    pub tier: i32,         // 1=Admin, 2=Dev, 3=Guest
}
```

**Memory** is distributed across several stores:
- `~/<agent_name>/memory/` — The agent's personal vault on disk: `SAVEUPS.md` (session retrospectives), `LEARNINGS.md` (technical lessons), `PONDERS.md` (open questions).
- `cass.db` — The CASS SQLite database holding `FactCard` (structured knowledge) and `EpisodicMemory` (AI-generated session summaries) records.

The Ghost exists whether or not an active Body is running.

### The Body (The Shell Session)

The Body is the execution environment: a terminal, a shell process, a working directory. It has two identifiers:

- **`body_id`**: A UUID generated fresh at each boot that uniquely identifies this specific terminal instance. Two boots of the same agent produce two different `body_id`s.
- **`session_id`**: Assigned by the Citadel when the lease is created. This is the Citadel's identifier for the session and is used to validate all subsequent gRPC calls.

The Body is temporary by nature. Its lifetime is bounded by the shell process and kept alive by periodic heartbeats. When the heartbeat stops, the TTL on the session's Redis lease key expires, and the session goes dark.

### The Fusion: `koad-agent boot`

The boot process is the act of fusing a Ghost to a Body. When `eval $(koad-agent boot --agent Tyr)` runs:

1. `koad-cli` reads Tyr's Ghost from `config/identities/tyr.toml`
2. It generates a fresh `body_id` (UUID)
3. It calls `CitadelSessionService::create_lease()` with the agent name and `body_id`
4. The Citadel checks: is there already an active session for "Tyr"? If yes, the request is rejected. If no, it creates a `SessionRecord` in Redis and returns a `session_id` and `session_token`
5. `koad-cli` prints `export KOAD_SESSION_ID="..."` and `export KOAD_SESSION_TOKEN="..."` to stdout
6. The `eval` wrapper executes these exports in the parent shell

After this, the shell *is* the Body. The `KOAD_SESSION_TOKEN` is the proof that this Body is authorized to act on behalf of the Tyr Ghost.

### Session Separation: `session_id` vs `body_id`

These two IDs serve different purposes:

| Identifier | Scope | Purpose |
|---|---|---|
| `body_id` | Client-side | Uniquely identifies the terminal/process instance. Generated by `koad-cli`, opaque to the Citadel after lease creation. |
| `session_id` | Citadel-side | The Citadel's reference for the active session. Used as the Redis key and carried in the `KOAD_SESSION_TOKEN`. |

The `session_id` is what the interceptor validates. The `body_id` is stored in the `SessionRecord` so the Citadel can tell *which terminal* this session came from, useful for audit logs.

### Session End: The Ghost Lives On

When a session ends — either via `koad logout` (calling `CloseSession`) or via timeout/crash — the Body is destroyed:
- The `SessionRecord` is removed from Redis
- The `koad:session:<session_id>` lease key is deleted
- A `session_closed` event fires on the Koad Stream

But the Ghost is unaffected. The TOML identity file still exists. The memory in `cass.db` and the personal vault on disk persists intact. CASS's `EndOfWatchPipeline` receives the `session_closed` event and generates a summary of the session's activities, which it saves to the `episodes` table in `cass.db` — so the Ghost actually *grows richer* after each session ends.

On the next boot, the Ghost inhabits a new Body with full access to everything it learned from all prior sessions.

## Failure Modes & Edge Cases

**What happens if I try to boot the same agent in two terminals simultaneously?**
The second boot attempt will be rejected by `CitadelSessionService::create_lease()`. The Citadel scans active sessions in Redis before creating a new lease. If it finds an existing `active` session for the same agent name, it returns an error. The second terminal's `eval $(koad-agent boot ...)` will fail with a message indicating the agent is already active. To force a new session (e.g., after a crash), the old session must be purged first, either by waiting for the reaper or manually clearing it.

**What if the shell crashes without logging out?**
The Body dies, but the heartbeat stops reaching the Citadel. After `lease_duration_secs` (configured in `kernel.toml`), the TTL on the Redis lease key expires and the session goes dark. The Citadel's reaper task will eventually detect this and purge the session, then fire the `session_closed` event so CASS can summarize it. The "One Body, One Ghost" lock is released when the purge completes, allowing a new boot.

**Where is the agent's memory between sessions?**
The Ghost's memory exists in two places: the personal vault on disk (`~/<agent>/memory/`) and the `cass.db` SQLite database. Both are durable storage that persist indefinitely. The personal vault is managed manually (agents write to it directly); `cass.db` is managed by the CASS service.

## FAQ

### Q: What happens if I try to boot the same agent in two different terminals?
The Citadel rejects the second boot. `CitadelSessionService::create_lease()` checks for an existing active session for the agent name before creating a new one. The enforcement code lives in `crates/koad-citadel/src/services/session.rs`. If you need to force a new session after a crash, wait for the reaper to purge the dark session (controlled by `dark_timeout_secs` in `kernel.toml`) or ask the Admiral to manually clear it.

### Q: Where is my agent's memory stored when it's not running?
In two places: the agent's personal vault on disk at `~/<agent_name>/memory/` (flat files like `SAVEUPS.md`) and the CASS SQLite database at `~/.koad-os/cass.db` (structured `FactCard` and `EpisodicMemory` records). Both persist indefinitely regardless of session state.

### Q: What's the difference between a `session_id` and a `body_id`?
The `body_id` is a client-generated UUID that identifies the specific terminal instance — it's generated by `koad-cli` before contacting the Citadel. The `session_id` is Citadel-assigned upon lease creation and serves as the Citadel's primary key for the session in Redis. The `session_id` is what gets embedded in the `KOAD_SESSION_TOKEN` and validated by the gRPC interceptor on every call.

### Q: How does the system remember who I am between sessions?
The Ghost's identity never changes — it's a TOML file on disk. The memory that *accumulates* between sessions is written to `cass.db` via the CASS `MemoryService` (for `FactCard`s committed during a session) and the `EndOfWatchPipeline` (which auto-generates an `EpisodicMemory` summary when a session closes). On next boot, CASS's `HydrationService` reads those records and builds a context packet that contains the agent's recent history and high-significance facts.

### Q: Why is it called the "Body/Ghost" model?
The metaphor maps to the practical reality of how agents exist. An AI agent's identity and accumulated knowledge outlives any single shell session — just as a Ghost is persistent and immaterial. The shell session is the temporary physical vessel the Ghost inhabits to do work. When the shell dies, the Ghost remains. When a new shell boots, the Ghost returns. The naming makes the separation of concerns explicit and memorable.
