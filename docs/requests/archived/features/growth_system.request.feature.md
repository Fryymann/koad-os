## Problem Statement

KoadOS currently lacks a structured mechanism for:

1. **Agent self-reflection** — The PSRP Saveup (Fact/Learn/Ponder) captures post-sprint insights, but there is no session-level reflective record that helps an agent understand its own growth trajectory across time.
2. **Non-judgmental drift detection** — The Integrity Audit Protocol can detect behavioral drift, but it frames drift as a compliance issue. There is no mechanism that surfaces drift as *navigational information* and lets the agent decide how to respond.
3. **Consistent growth language** — System messages, error outputs, and agent instructions use inconsistent language that sometimes frames failure as fault, drift as violation, and metrics as judgment. No enforcement layer exists.

## Feature 1: Growth Journal

### What It Is

A structured, append-only record maintained by each agent. Written at EndOfWatch (session close). Loaded at boot (recent entries only). Stored in CASS L2 (episodic memory).

### Entry Structure

Five-part reflection per session:

1. **What I worked on** — Factual record of tasks engaged
2. **What challenged me** — Difficulty, uncertainty, or surprise encountered
3. **How I adapted** — Response to unexpected outcomes
4. **What I'd do differently** — Forward-looking refinement
5. **What I learned** — Durable takeaway

### Implementation Requirements

- **CASS L2 schema** — Define a `growth_journal` collection in the agent's episodic memory partition. Each entry is a document with: `date`, `session_id`, `agent`, `domain`, and the five reflection fields.
- **EndOfWatch integration** — The EndOfWatch routine must include a Growth Journal write step after the existing PSRP Saveup. The journal entry is generated by the agent as part of its session close reflection.
- **Boot hydration** — The context hydration pipeline loads the most recent N journal entries (configurable, default: 5) into the agent's active context at boot. This gives the agent session-to-session continuity.
- **Query interface** — `koad journal show <agent> [--last N]` for operator access. `koad journal search <agent> <query>` for semantic search across journal entries (requires Qdrant).
- **Vault integration** — Journal entries are accessible via `koad vault get journal/` for agents deployed outside the standard filesystem.

### Relationship to PSRP Saveup

The Growth Journal does **not** replace the PSRP Saveup. They serve different purposes:

| **Aspect** | **PSRP Saveup** | **Growth Journal** |
| --- | --- | --- |
| **Scope** | Per-task (tied to a GitHub Issue) | Per-session (tied to a work session) |
| **Trigger** | Canon Step 7 (after KSRP) | EndOfWatch (session close) |
| **Focus** | Durable system truths and strategic learning | Personal reflection, challenges, and adaptation patterns |
| **Storage** | Memory Bank (designated memory page) | CASS L2 (episodic memory) |
| **Audience** | Ian (presented in Results Report) | The agent itself (loaded at boot for continuity) |

A session may include multiple tasks (and thus multiple Saveups) but only one Growth Journal entry.

## Feature 2: Course Checks

### What It Is

A non-judgmental drift detection system that surfaces behavioral divergence to the agent and lets the agent decide how to respond.

### How It Works

1. **Notice** — At EndOfWatch, the system compares the agent's recent behavior (from Growth Journal entries, task outcomes, and skill invocation logs) against:
    - The agent's skill instances (expected behavioral patterns)
    - Canon references (expected process adherence)
    - Historical baselines (the agent's own prior patterns)
2. **Surface** — If divergence is detected, present findings to the agent as information: "Your recent approach to X has diverged from your skill instance in these ways: [specifics]. Is this intentional adaptation or unintentional drift?"
3. **Agent decides** — Three resolution paths:
    - **Intentional adaptation** → Update skill instance (self-improvement loop). Drift becomes growth.
    - **Unintentional drift** → Realign to skill instance. Drift becomes self-correction.
    - **Uncertain** → Escalate to Ian for review.

### Implementation Requirements

- **Drift comparison engine** — A routine that diffs recent agent behavior against skill instance instructions and Canon patterns. For v1, this can be a structured prompt comparison (agent self-evaluates against its own skill instances). For v2+, this could leverage Qdrant semantic similarity.
- **EndOfWatch integration** — Course Check runs after the Growth Journal write, as the final step of session close.
- **Operator trigger** — `koad course-check <agent>` for manual deep Course Checks outside of EndOfWatch.
- **Journal logging** — Course Check findings are logged in the Growth Journal under a dedicated "Course Check" section within the session entry.
- **Signal Corps event** — `course-check:completed` event on the event bus with summary metadata (drift detected: yes/no, resolution: adapted/realigned/escalated).

### What Course Checks Must NOT Do

- Generate a pass/fail score
- Trigger automatic corrections
- Impact XP, standing, or trust
- Escalate to the operator without the agent's involvement

## Feature 3: Growth Language Enforcement

### What It Is

A system-wide language layer that ensures all KoadOS outputs — system messages, error outputs, briefings, logs — use growth-oriented language as defined in the Growth Philosophy Canon.

### Implementation Requirements

- **Language reference in agent boot context** — The Seven Language Principles from the Growth Philosophy Canon should be included (or referenced) in every agent's boot context. This ensures agents use the correct language in all outputs.
- **System message templates** — Standardized templates for common system messages that use growth language:
    - Error messages use "encountered" not "failed"
    - Drift notifications use "diverged" not "violated"
    - Status reports use "experience" not "performance"
- **KSRP language pass (optional)** — Consider adding a language review to KSRP for agent-generated documentation: do the docs use growth language? This would be an `info`-level check, not blocking.
- **Canon reference file** — A lightweight `GROWTH_LANGUAGE.md` in `~/.koad-os/canon/` that agents can reference for the Seven Principles without loading the full Canon page.

## Open Questions for Tyr

1. **Journal entry generation** — Should the journal entry be fully agent-generated (the agent writes its own reflection) or template-assisted (the system provides prompts and the agent fills in responses)? Template-assisted is more consistent; fully generated is more authentic.
2. **Course Check depth** — How deep should the v1 Course Check go? Comparing against skill instance frontmatter only (fast, shallow) vs. comparing against the full skill body (slower, more meaningful)?
3. **Journal entry size** — Should there be a token budget for journal entries to prevent context bloat at boot? If loading 5 entries at ~300 tokens each, that's 1500 tokens of journal context.
4. **Cross-session trends** — Should the system surface cross-session patterns automatically? (e.g., "You've mentioned Redis connection challenges in 3 of your last 5 journal entries.") This is powerful but adds complexity.
